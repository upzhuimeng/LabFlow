LabFlow 智序 - 设计/开发难点分析

===============================================================================

1. AI 结构化输出不稳定问题

在智能预约助手的开发过程中，最具挑战性的问题之一是大语言模型的结构化输出不稳定。系统期望模型返回符合 AgentOutput Schema 的 JSON 结构，包含实验室信息（lab_id、lab_name、address）、时间段（start_time、end_time）、推荐理由（reason）等字段。然而，实际运行中发现模型输出存在几种不稳定情况：返回的 JSON 格式正确但字段值不符合预期；返回的文本无法解析为有效的 JSON；模型直接返回纯文本描述而非结构化数据。

针对这一问题，团队实现了多层次的 Fallback 机制。第一层尝试按标准 JSON 解析；第二层尝试从文本中提取关键信息进行解析；第三层在所有解析失败时，调用 build_suggestion_from_db() 函数直接查询数据库获取可用资源推荐。这种多层次的容错设计确保了智能助手在任何情况下都能给出有意义的回复，虽然推荐结果可能不如 AI 解析的结果智能，但至少保证了功能的可用性。

===============================================================================

2. 异步编程与事务管理

LabFlow 采用 FastAPI + SQLAlchemy 2.0 的全异步架构，这带来了性能优势，但也增加了开发复杂度。在异步上下文中管理数据库事务需要特别注意 session 的生命周期。在预约创建流程中，需要在同一个事务内完成预约记录创建、日志记录、发送初始通知等多个操作，如果其中任何一步失败，整个事务需要回滚。

为了简化异步事务处理，团队定义了 get_db() 依赖注入函数，为每个请求提供独立的数据库会话。会话在请求结束时自动关闭，确保资源正确释放。对于需要手动控制事务的场景，使用 async with 上下文管理器显式管理提交和回滚。

===============================================================================

3. 前端权限控制与状态管理

前端权限控制面临的主要挑战是：实验员角色只能看到和操作自己的数据，而管理员可以看到所有数据。在 Next.js 的服务端渲染环境下，需要在服务端完成权限验证，确保未授权的用户无法通过 API 获取他人的数据。

解决方案分为接口层和数据层两个维度。接口层通过自定义装饰器验证用户角色，确保调用者有权限访问特定接口。数据层在 SQL 查询时自动添加用户 ID 过滤条件，例如实验员查询预约列表时，查询语句自动附加 user_id = current_user.id 条件。两种维度结合，确保了权限控制的完整性。

===============================================================================

4. 软删除与级联处理

LabFlow 采用了软删除策略，所有数据删除操作都不物理删除记录，而是修改 status 或 is_active 字段。这种设计保留了历史数据的完整性，支持数据统计和审计查询，但也带来了级联处理的复杂性。

以删除实验室为例，当管理员删除一个实验室时，需要同时处理关联的仪器数据、预约数据、用户关联数据等。如果直接将这些关联数据的 status 也标记为删除，可能影响用户正在使用的预约记录；如果保留关联数据，又会造成数据不一致。

团队采用的方案是：删除实验室时，将实验室状态标记为停用，关联的仪器一并标记为停用，但保留预约记录的原状态。这样既保证了历史预约的可追溯性，又确保了已删除的实验室在界面上不再可见。用户在查询可用实验室时，系统自动过滤掉状态为停用的记录。

===============================================================================

5. 通知系统的异步处理

通知系统需要支持多场景的自动触发，包括预约提交、审批完成、预约取消、仪器维护等。通知发送采用异步后台执行，调用 AI 总结等耗时操作时，用户界面立即返回成功提示，实际的通知发送在后台完成。通知数据存储包含类型（type）、关联ID（related_id）和附件数据（attachment）字段，其中 attachment 以 JSON 格式存储智能推荐等场景的完整上下文信息。

===============================================================================

6. 卡片附件与消息联动

卡片附件是 LabFlow 的一个创新设计，但也带来了开发复杂度。传统的通知只包含文字内容，而卡片附件需要存储完整的上下文数据，并在消息详情页以结构化的方式展示。

实现上，每条通知包含类型（type）和关联ID（related_id）字段，标识这条通知关联的业务对象。消息详情页根据类型调用相应的 API 获取完整数据，渲染为卡片形式展示。例如，审批结果通知（type=1）的 related_id 指向 approval 记录，详情页通过 approval_id 获取审批的完整信息，包括预约详情、审批理由等。

这种设计的优势在于保持了数据的规范性，避免了在通知表中存储大量冗余字段。同时，关联的业务数据变更后，消息卡片展示的内容也会同步更新。
